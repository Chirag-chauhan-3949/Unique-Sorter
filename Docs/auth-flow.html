Authentication Flow

How Registration and Login work in Unique Sorter

Technologies Used

Next.js 16 (API Routes) Firebase Admin SDK (Server) Cloud Firestore (Database) JWT (jsonwebtoken) In-Memory DB (Fallback) React (Frontend)

Key Point

All database operations use the Firebase Admin SDK (server-side), NOT the Firebase Client SDK. This means:

  • No reCAPTCHA needed
  • No client-side Firebase Auth
  • All Firestore reads/writes happen on the server via API routes
  • 5-second timeouts on all Firebase calls to prevent hanging

Registration

User creates an account with name, phone, password

Visual Flow

User (Browser) Server Firebase ============ ====== ======== /register page Fill: Name, Phone, Password, Confirm Password | |--- POST /api/auth/register ----->| | { name, phone, password } | | |--- Check: is role "admin"? | | YES -> Return 403 (blocked) | | | |--- adminDb.collection('userdata') | | .doc(phone).get() ------------>| Firestore | | | Check if | |<--- exists? ----------------------| phone exists | | YES -> Return 409 (duplicate) | | | |--- db.createUser() (local memory) | | | |--- adminDb.collection('userdata') | | .doc(phone).set({...}) ------->| Firestore | | (fire & forget, don't wait) | Save user | | |<--- 200 { success, user } -------| | Show "Registration Successful!" Button: "Go to Login"

Step by Step

1 User fills the registration form
Fields: Full Name, Phone Number (+91), Password, Confirm Password
Role dropdown exists but admin is blocked server-side — always registers as "user"
2 Frontend sends POST /api/auth/register
Sends: { name, phone, password, role }
File: src/app/register/page.js line 52
3 Server validates input
Checks:
- Name, phone, password are present
- Phone matches regex /^[6-9]\d{9}$/ (Indian 10-digit)
- Role is NOT "admin" (returns 403 if it is)
4 Check if phone already exists in Firestore
Uses Firebase Admin SDK to query userdata collection
Has a 5-second timeout — if Firestore is slow, it skips the check
If phone exists → returns 409 "already exists"
5 Create user in local in-memory DB
Creates user object with Date.now() as ID
Stored in src/lib/db.js — lost on server restart
6 Sync to Firebase Firestore (fire and forget)
Writes to userdata/{phone} collection in Firestore
Does NOT wait for Firestore response — returns success immediately
Sync happens in background
7 Return success to frontend
Returns: { success: true, user: { id, name, phone, role } }
Frontend shows success screen with "Go to Login" button

Step by Step — Send OTP

1 User enters phone number on login page
Just the phone number — no password, no role needed
2 Frontend sends POST /api/auth/send-otp
Sends: { phone: "7999952226" }
3 Server checks if phone is registered
Queries Firestore userdata/{phone} using Admin SDK (5s timeout)
Falls back to local in-memory DB if Firestore fails
If NOT found → returns 404 "Please register first"
4 Generate 6-digit OTP
Math.floor(100000 + Math.random() * 900000)
Stored in: in-memory Map + Firestore otps/{phone}
Expires in: 5 minutes
5 OTP delivery (currently: server console only)
WORKING OTP printed in server terminal
TODO Send via SMS (needs SMS provider like Fast2SMS, MSG91, or Twilio)
The OTP is NOT sent to the user's phone yet — this needs an SMS API integration

Step by Step — Verify OTP

1 User enters 6-digit OTP
6 individual input boxes with auto-focus, paste support, backspace navigation
2 Frontend sends POST /api/auth/verify-otp
Sends: { phone: "7999952226", otp: "123456" }
3 Server verifies OTP
Checks in-memory Map first, then Firestore otps/{phone}
Validation:
- OTP not found → 400 "Request a new one"
- OTP expired (>5 min) → 400 "Expired"
- Wrong OTP → 401 "Invalid" (max 3 attempts)
- 3 failed attempts → 429 "Too many attempts"
4 OTP is correct — delete it
Removes OTP from memory Map AND Firestore otps/{phone}
OTP is single-use — cannot be reused
5 Fetch user data from Firestore
Queries userdata/{phone} to get name, role, etc.
6 Generate JWT token
Uses jsonwebtoken library
Payload: { userId, phone, role }
Secret: JWT_SECRET from .env.local
Expires in: 24 hours
7 Return token + user to frontend
Response: { token, user: { id, name, phone, role } }
Frontend stores token in localStorage
Redirects to /dashboard

Security Features

What's Working

  • Admin registration blocked — users cannot self-register as admin
  • OTP expires in 5 minutes — cannot be reused
  • Max 3 OTP attempts — brute force protection
  • JWT tokens with 24h expiry — signed with secret key
  • Phone format validation — Indian 10-digit numbers only
  • 5-second timeouts — Firebase calls never hang
  • Fire-and-forget writes — registration never blocks
  • Dual OTP storage — in-memory + Firestore backup

What Needs Work (Production)

  • SMS delivery — OTP only prints to console, needs SMS API (Fast2SMS/MSG91/Twilio)
  • Password hashing — passwords stored as plain text, needs bcrypt
  • Rate limiting — no limit on how many OTPs can be requested
  • Middleware — no server-side route protection, only client-side
  • HTTPS — required for production deployment
