Overview

Firebase is used for two things in this project:

  1. Phone Number Authentication (OTP) — Firebase sends a real SMS one-time password to the user's phone. No username/password is used.
  2. Firestore Database — All CRM data (users, enquiries, quotations, audit logs, OTP sessions) is stored in Cloud Firestore.

Firebase Admin SDK (Server)

  • Runs in Next.js API routes only
  • Initialized with a service account
  • Full access — bypasses security rules
  • Used to verify Firebase ID tokens
  • Used for all Firestore reads/writes
  • Never sent to the browser

Firebase Client SDK (Browser)

  • Runs in the browser only
  • Initialized with public API keys
  • Obeys Firestore security rules
  • Used to initialize the Auth object
  • Not used directly for phone auth flow
  • Loaded via src/lib/firebase.js
Key Design Decision OTP sending is done entirely server-side via the Firebase Auth REST API. This means no reCAPTCHA is required — the server calls Firebase directly as a trusted party.

SDKs and Initialization

Client SDK — src/lib/firebase.js

Initializes the Firebase app for use in the browser. Used only for setting up the auth, db, and storage objects. The phone auth flow itself goes through our own API routes, not the client SDK.

// Initializes once, reuses existing app on hot reload
app = getApps().length === 0 ? initializeApp(firebaseConfig) : getApps()[0];
db = getFirestore(app);
auth = getAuth(app);
storage = getStorage(app);

// Pre-warms reCAPTCHA Enterprise config (reduces first-auth latency)
// Only runs in browser, errors are silently ignored
initializeRecaptchaConfig(auth).catch(() => {});

Admin SDK — src/lib/firebase-admin.js

Initialized with a service account private key from environment variables. Only imported in src/app/api/ routes — never bundled to the browser.

// Exported objects used across all API routes:
adminAuth // firebase-admin/auth — verifyIdToken, getUser
adminDb   // firebase-admin/firestore — collection, doc, get, set, update, delete

OTP Authentication Flow

The complete login flow uses two API routes. No client-side Firebase SDK calls are made during authentication.

Step 1 — Send OTP (POST /api/auth/send-otp)

Step 1
Validate Input
Check phone format /^[6-9]\d{9}$/, rate limit (5/hour)
Step 2
Check User Exists
Look up userdata/{phone} in Firestore via Admin SDK
Step 3
Call Firebase REST API
accounts:sendVerificationCode with API key and phone number
Step 4
Store sessionInfo
Save Firebase sessionInfo token to otps/{phone} in Firestore (5min TTL)
Step 5
SMS Delivered
Firebase sends real SMS to user's phone via its carrier network
// Server calls Firebase Auth REST API — no reCAPTCHA needed server-side
POST https://identitytoolkit.googleapis.com/v1/accounts:sendVerificationCode?key=API_KEY
Body: { phoneNumber: '+919876543210', recaptchaToken: '' }

// Firebase responds with a sessionInfo token
Response: { sessionInfo: 'AJOnW4Q...' }

Step 2 — Verify OTP (POST /api/auth/verify-otp)

Step 1
Get sessionInfo
Fetch from Firestore otps/{phone}, check expiry (5 min)
Step 2
Verify OTP Code
Call accounts:signInWithPhoneNumber with sessionInfo + OTP code
Step 3
Verify ID Token
Firebase returns idToken → verified with adminAuth.verifyIdToken()
Step 4
Fetch User Data
Look up userdata/{phone} to get name, role
Step 5
Issue App JWT
Sign 24h JWT with userId, phone, role. Delete used OTP session.
// Verify the OTP the user typed
POST https://identitytoolkit.googleapis.com/v1/accounts:signInWithPhoneNumber?key=API_KEY
Body: { sessionInfo: 'AJOnW4Q...', code: '123456' }

// Firebase returns a Firebase ID token if OTP is correct
Response: { idToken: 'eyJhbGci...', localId: '...' }

// We verify it with Admin SDK, then issue our own JWT
const decoded = await adminAuth.verifyIdToken(firebaseData.idToken);
const token = jwt.sign({ userId, phone, role }, JWT_SECRET, { expiresIn: '24h' });
Why store sessionInfo in Firestore? Next.js runs multiple worker processes in development (Turbopack) and in production (serverless functions). An in-memory Map is not shared across workers — so the sessionInfo would be lost between the send and verify requests. Firestore solves this.

Two-Token System

This project uses two different tokens, each with a specific purpose:

TokenIssued ByLifetimePurposeStored In
Firebase ID Token Firebase Auth 1 hour Proves the user successfully verified their phone number via OTP. Used only during the login API call to exchange for our JWT. Not stored — used immediately then discarded
App JWT Our server (/api/auth/verify-otp) 24 hours Authorizes all subsequent API calls. Contains userId, phone, role. Verified by src/lib/auth.js. Browser localStorage + cookie (authToken)
Why not use Firebase ID tokens for API auth? Firebase ID tokens expire after 1 hour and require a network call to Firebase to verify. Our app JWT is verified locally (no network round-trip), expires after 24 hours for better UX, and contains our custom role field which Firebase doesn't know about.

Auth Header Format

// All protected API calls send:
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

// Verified in src/lib/auth.js:
const { userId, phone, role } = jwt.verify(token, JWT_SECRET);

Security Setup

Firebase Phone Auth Settings

  • Phone authentication is enabled in the Firebase Console under Authentication → Sign-in methods
  • SMS region policy is set to DENY with empty country list (meaning all countries are allowed, including India)
  • reCAPTCHA Enterprise is enabled in Google Cloud Console (required for client SDK, not for server-side REST API calls)

Firestore Security Rules

Because all database access goes through our server-side Admin SDK (which bypasses rules), Firestore security rules act as a defense-in-depth layer. The Admin SDK uses a service account with full access.

// firestore.rules — locks down direct client access
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    // Block all direct client access — use API routes instead
    match /{document=**} {
      allow read, write: if false;
    }
  }
}

Rate Limiting

Implemented in src/lib/rateLimit.js using an in-memory Map:

EndpointLimitWindow
Send OTP5 requestsPer hour, per phone number
Verify OTP5 attemptsPer hour, per phone number
Firebase token exchange10 requestsPer hour, per phone number
Production Note The in-memory rate limiter is not shared across multiple server instances (serverless). For production with multiple instances, replace with a Redis-based rate limiter or use Firebase Firestore as the backing store.

Service Account

  • Service account JSON downloaded from Firebase Console → Project Settings → Service Accounts
  • Key fields extracted and stored as individual environment variables (not the whole JSON)
  • Private key must have literal newlines in the env var, or use .replace(/\\n/g, '\n') when initializing
  • Never commit the service account JSON to git

What Firebase Does and Doesn't Do

CapabilityFirebase Does It?Notes
Send SMS with OTP to phone Yes Firebase contacts the carrier network and delivers a real SMS. Requires Blaze (pay-as-you-go) plan.
Generate the OTP code Yes Firebase generates a random 6-digit code internally. We never see this code — only the user does.
Verify the OTP the user types Yes We send the user's code to accounts:signInWithPhoneNumber and Firebase confirms if it matches.
Store user profile data (name, role) No Firebase Auth only stores phone number and UID. We store name, role, etc. in Firestore userdata.
Issue long-lived session tokens No Firebase ID tokens last 1 hour. We issue our own 24h JWT after verification.
Enforce access control (RBAC) No Role-based access (USER vs ADMIN) is managed by our API routes using the JWT role field.
Handle reCAPTCHA for server calls No reCAPTCHA is only required for client-side Firebase SDK calls. Server-to-server REST API calls do not need reCAPTCHA.
Deliver SMS to Indian numbers Yes Requires the SMS region policy to include India (or allow all regions by setting DENY with empty list).
Summary Firebase handles the hard part — SMS delivery and OTP verification. Our server handles user data, sessions, access control, and business logic. The two systems are connected at exactly one point: when we exchange a Firebase ID token for our own app JWT.
