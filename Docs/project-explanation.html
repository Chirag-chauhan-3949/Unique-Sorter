Unique Sorter

A business management web app for a sorting machine company

What is this project?

Unique Sorter is a web application built for a company that sells sorting machines (like color sorters). It helps the company manage:

  • Customer Inquiries - People interested in buying machines fill out a form
  • Quotations - The company creates price quotes for customers
  • Clients - Keep track of all customers
  • Orders - Manage machine orders
  • Inventory - Track products in stock
  • Reports - View business data

Tech Stack

Next.js 16 React Firebase / Firestore Tailwind CSS ShadCN UI Axios

How Does It Work?

1. Login

User opens /login -> Enters phone number + password -> Server checks if user exists in Firebase -> If valid, server generates a 6-digit OTP -> User enters OTP -> Server verifies OTP, creates a token -> User is logged in and sent to Dashboard

2. Public Inquiry Form

Customer visits /enquiry (public page) -> Fills in: name, phone, email, company, product interest, message -> Clicks Submit -> Data goes to Backend API -> saved to Firebase -> Customer sees success message

3. Dashboard (Admin)

Admin logs in -> sees Dashboard at /dashboard -> Can view: Enquiries, Clients, Quotations, Orders, Inventory, Reports, Settings -> Can create quotations for customers -> Can manage everything from one place

Critical Problems

Critical

1. Passwords Stored as Plain Text

User passwords are saved without any hashing. Anyone with database access can read all passwords.

src/app/api/auth/register/route.js - stores password directly

src/app/api/auth/verify-credentials/route.js - compares plain text

src/lib/db.js - plain text comparison

// Current (INSECURE):
if (user.password !== password) return null;

// Should be:
const match = await bcrypt.compare(password, user.hashedPassword);

Note: bcryptjs is already installed in package.json but never used!

Critical

2. Fake Token - No JWT Signing

The auth token is just base64-encoded JSON. Anyone can forge it - no signature, no secret key.

src/app/api/auth/verify-credentials/route.js:6-15

// Current (INSECURE):
return Buffer.from(JSON.stringify(payload)).toString('base64');

// Should use jsonwebtoken:
return jwt.sign(payload, process.env.JWT_SECRET, { expiresIn: '24h' });
Critical

3. No Middleware - Routes Are Unprotected

There is no middleware.js file. Any user can directly access /dashboard, /api/enquiry, /api/quotations without being logged in.

Auth is only checked on the client side (JavaScript) which can be easily bypassed.

Missing file: src/middleware.js does not exist

Critical

4. API Routes Have No Auth Checks

None of the API endpoints verify the user's token before responding:

  • /api/enquiry - anyone can read/write enquiries
  • /api/quotations - anyone can read/write quotations
  • /api/inquiries - anyone can CRUD all inquiries
Critical

5. Missing .env.local File - Firebase Won't Connect

No environment file exists. The app needs these variables to connect to Firebase:

NEXT_PUBLIC_FIREBASE_API_KEY=
NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN=
NEXT_PUBLIC_FIREBASE_PROJECT_ID=
NEXT_PUBLIC_FIREBASE_STORAGE_BUCKET=
NEXT_PUBLIC_FIREBASE_MESSAGING_SENDER_ID=
NEXT_PUBLIC_FIREBASE_APP_ID=
FIREBASE_ADMIN_PROJECT_ID=
FIREBASE_ADMIN_CLIENT_EMAIL=
FIREBASE_ADMIN_PRIVATE_KEY=

Without this file, the entire database layer silently fails.

Critical

6. No Role Validation on Registration

Anyone can register as admin by sending "role": "admin" in the request body. There is no server-side check.

src/app/api/auth/register/route.js:7

// Current - accepts any role:
const { name, phone, password, role = 'user' } = await request.json();
// No validation that role is actually "user"!

High Priority Problems

High

7. Inconsistent Role Format

Roles are stored differently across the codebase:

  • register/route.js - stores as 'user' (lowercase)
  • verify-credentials/route.js - converts to 'USER' (uppercase)
  • firebaseSync.js - stores as-is (no normalization)

This causes role checks to fail randomly.
